First-Class Agent Hosts

Tier 3 design for rebuilding c3spec around Cursor, Claude Code, and Codex.

1. Product Shape

c3spec will support exactly three first-class agent hosts for now: Cursor, Claude Code, and Codex. The existing long-tail host adapters will be removed until there is a real need to support them with the same first-class standard.

Users should be able to enter a similar natural-language prompt in any supported host, such as "Use c3spec to plan and implement support for X." Slash commands are removed from the core workflow.

The c3spec workflow explicitly asks the host to use named subagents for implementation and review. This satisfies Codex's explicit subagent workflow requirement and keeps Cursor and Claude behavior consistent rather than relying on implicit delegation.

2. Canonical Architecture

The source of truth becomes .agents/. Cursor and Codex read .agents/skills/ directly. Claude cannot, so generation materializes a copy into .claude/skills/. No symlinks are used. 

.agents/
  skills/
    c3spec-start/SKILL.md
    c3spec-tier1-fix/SKILL.md
    c3spec-tier2-feature/SKILL.md
    c3spec-subagent-dev/SKILL.md
    c3spec-host-adapter/SKILL.md
  agents/
    implementer.yaml
    spec-reviewer.yaml
    quality-reviewer.yaml
  hooks/
    session-start.yaml
    memory-scan.js

Canonical agent manifests render into native host formats:

Host Generated Agents Notes
Cursor .cursor/agents/<name>.md Markdown agents with YAML frontmatter.
Claude Code .claude/agents/<name>.md Markdown agents with Claude-specific frontmatter.
Codex .codex/agents/<name>.toml TOML agents with name, description, and developer_instructions.

The c3spec-host-adapter skill explains host quirks at runtime. Other skills name the role to dispatch, such as spec-reviewer, and the host adapter maps that role to the current host's native subagent mechanism.

3. Host Behavior

Cursor

Cursor consumes canonical skills from .agents/skills/. c3spec generates .cursor/agents/, .cursor/hooks.json, and optional rule files. It does not generate .cursor/skills/ for now, to avoid a second skill surface drifting from the canonical source.

Claude Code

Claude gets generated .claude/skills/, .claude/agents/, .claude/settings.json, and CLAUDE.md. The generated skill copies are derived artifacts and should be protected by drift detection.

Codex

Codex consumes canonical skills from .agents/skills/. c3spec generates .codex/agents/, .codex/config.toml, .codex/hooks.json, and AGENTS.md. Codex config should pin sane agent defaults: 

[agents]
max_threads = 6
max_depth = 1

max_depth = 1 matches c3spec's controller model: the root orchestrator can spawn implementer and reviewer agents, but workers should not recursively spawn more workers.

4. Generator and Update Flow

init, sync, and update all run the same generation pipeline. 

canonical input
  .agents/skills/*
  .agents/agents/*.yaml
  .agents/hooks/*

pipeline
  validate canonical artifacts
  render host-specific artifacts
  compare generated hashes
  write safe outputs
  report drift / hand edits

init creates canonical artifacts if missing, then renders host outputs. sync re-renders host outputs from canonical artifacts. update refreshes bundled canonical artifacts and regenerates host outputs with warnings for local canonical edits or hand-edited generated files.

The old command-adapter model should be replaced or narrowed into host-generation adapters. The new adapter contract centers skills, agents, hooks, instructions, and config, not slash command files.

5. Implementation Stages

  1. Remove unsupported hosts: delete non-Cursor/Claude/Codex adapters, tool entries, tests, and docs.
  2. Add canonical model: validate .agents/skills, agent manifests, and hook manifests.
  3. Build host renderers: generate Cursor markdown agents/hooks, Claude skills/agents/settings, and Codex TOML/config/hooks.
  4. Wire commands: make init, sync, and update use the shared pipeline with drift protection.
  5. Dogfood and document: regenerate this repo, update docs/specs, run tests/build, and capture memory/retro.

Verification includes TypeScript compile, unit and snapshot tests for every host output, TOML/JSON/frontmatter parsing tests, end-to-end temp project tests for init/sync/update, and a final dogfood pass on this repository.

Open Detail Questions for Planning

  • Host detection mechanism for c3spec-host-adapter.
  • Generated-file sentinel/hash format and overwrite policy.
  • Memory-scan hook portability: Node script versus shell scripts per platform.
  • Whether Cursor rules are needed immediately or can wait until hooks/skills prove sufficient.
