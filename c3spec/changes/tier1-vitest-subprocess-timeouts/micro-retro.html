1.Why did this bug get introduced?

The vitest defaults (testTimeout: 10000, hookTimeout: 10000) carried over from project init never accounted for the test pattern that test/helpers/run-cli.ts actually uses — spawning dist/cli/index.js as a real Node subprocess for every CLI invocation. Each subprocess costs ~0.5–1.5s of cold-start (Node + Commander + the project's entire import graph) before any test logic runs. Tests that stack 5–10 runCLI calls reliably exceeded the 10s budget under realistic CPU contention, but the failure was load-dependent enough to look flaky rather than structural — so the timeout never got revisited.

2.What class of bug is this?

Configuration calibration mismatch — default value silently invalidated by test pattern.

A framework default optimized for one assumption (in-process unit tests) fails predictably when the actual test pattern violates that assumption (subprocess-per-assertion CLI tests). Adjacent class: bandaid hides latency. The fix is a bandaid because the real cost (subprocess startup × N) is still being paid — we just stop calling it failure.

Recognizable next time when: a test helper spawns external processes, a config sets a per-test or per-hook budget, and the helper is called in a loop or multiple times per test. Same pattern would surface for any future test using spawn, execFileSync, Docker, or a remote service mock.

3.What would prevent the next one?

Both:

  • Memory entry — capture the constraint “subprocess-based CLI tests need generous vitest timeouts” under c3spec/memory/constraints/vitest-subprocess-cli-tests.md so any future test author or vitest config tweaker hits this learning before tripping over it.
  • IDEAS #18 — refactor runCLI to in-process so the bandaid becomes unnecessary and tests cost milliseconds instead of seconds.
