Design: c3spec Bootstrap

2026-05-25  ·  tier2-c3spec-bootstrap  ·  4 decisions

Context

Forking @fission-ai/openspec → c3spec. The source is TypeScript (~779-line init command, centralized config in src/core/config.ts, PostHog telemetry in src/telemetry/). Key findings from source inspection:

  • src/core/config.ts exports OPENSPEC_DIR_NAME = 'openspec' — the intended central constant — but ~15 files hardcode 'openspec' directly (list.ts, archive.ts, view.ts, specs-apply.ts, etc.)
  • src/core/init.ts has DEFAULT_SCHEMA = 'spec-driven' (local constant) and profile system with two values: 'core' | 'custom'
  • Telemetry is a full PostHog integration with consent flow, UUID generation, CI auto-disable
  • No existing memory command — clean addition point in src/commands/

Decisions

D1 Directory rename strategy: central constant vs. global find-replace

Chosen

Global find-replace and rename the constant. Change OPENSPEC_DIR_NAME = 'openspec' → C3SPEC_DIR_NAME = 'c3spec' in config.ts, and replace all ~15 hardcoded 'openspec' path strings across the codebase to 'c3spec'.

// src/core/config.ts
export const C3SPEC_DIR_NAME = 'c3spec';

// Files that hardcode the path:
// list.ts, archive.ts, view.ts, specs-apply.ts,
// planning-home.ts, project-config.ts, resolver.ts,
// instruction-loader.ts, change-utils.ts

Alternatives considered

Central constant only — rename OPENSPEC_DIR_NAME and rely on TypeScript compile errors to catch usages. Rejected: many files don't import the constant at all and use string literals directly; compile errors won't catch them.

Runtime config (read dir name from config.yaml) — make the directory name configurable. Rejected: over-engineering for a clean-break fork. We own the name, it doesn't need to be a variable.

Reason

Both steps are needed. The constant rename is the "right" fix. The find-replace catches the ~15 files that never imported the constant. Together they guarantee no orphan 'openspec' path strings survive. A global grep after the change verifies completeness.

D2 Telemetry removal: no-op stubs vs. full deletion

Chosen

Replace src/telemetry/index.ts with a minimal no-op module that exports the same function signatures. Remove posthog-node from package.json. Strip the consent prompt from init.ts.

// src/telemetry/index.ts (replacement)
export async function maybeShowTelemetryNotice() {}
export async function trackCommand(
  _cmd: string, _ver: string
) {}
export async function shutdown() {}

Alternatives considered

Delete telemetry entirely — remove all import sites too. Rejected: the CLI entry point (src/cli/index.ts) calls maybeShowTelemetryNotice, trackCommand, and shutdown in hooks. Deleting requires editing those hooks, which risks breaking the command lifecycle. No-ops are zero-risk.

Keep telemetry, set own PostHog project — replace the API key. Rejected: no value in tracking your own private tool usage.

Reason

No-ops preserve the call sites in cli/index.ts untouched. The only required changes are: (1) replace src/telemetry/index.ts content, (2) remove posthog-node from package.json, (3) remove the first-run consent prompt from init.ts. Surgical and verifiable.

D3 c3spec init: extend profile system vs. change defaults

Chosen

Change the defaults, don't extend the profile system. Two targeted changes:

  1. Change DEFAULT_SCHEMA = 'spec-driven' → 'superpowers-bridge' in init.ts
  2. Add a post-scaffold step at the end of InitCommand.run() that creates the memory structure and copies the c3spec skills
// Added to InitCommand.run() after config.yaml written:
await scaffoldC3specStructure(openspecPath, projectPath);

// scaffoldC3specStructure creates:
//   c3spec/memory/MEMORY.md
//   c3spec/memory/{bug-patterns,workflow,constraints,design-decisions}/
//   .cursor/skills/openspec-{start,tier1-fix,tier2-feature,subagent-dev}/
//   Appends c3spec routing block to CLAUDE.md (or creates it)

Alternatives considered

Add a new 'c3spec' profile to the profile system — 'core' | 'custom' | 'c3spec'. Rejected: the profile system controls which workflow commands to install, not directory scaffolding. Wrong abstraction layer.

Separate c3spec scaffold command — run after init. Rejected: the whole point is one command bootstraps everything. c3spec init should be the answer.

Reason

The memory structure and skills are not profile-configurable concepts — they're always part of c3spec. Adding them as a post-scaffold step inside InitCommand.run() is the path of least resistance: no new abstractions, straightforward to test, and the skills content lives in the package's schemas/superpowers-bridge/ tree as source files.

D4 c3spec memory command: file structure and promote semantics

Chosen

New file src/commands/memory.ts, registered in src/cli/index.ts. Three subcommands:

  • list — reads c3spec/memory/MEMORY.md, prints index with category grouping and file sizes
  • add <category> <slug> — creates c3spec/memory/<category>/<slug>.md with correct frontmatter template and appends a pointer line to MEMORY.md
  • promote <slug> — searches retro files for - [ ] ... <slug> patterns, verifies the corresponding memory file exists, marks it - [x]
// c3spec/memory/workflow/my-learning.md frontmatter:
---
name: my-learning
description: one-line summary
metadata:
  type: workflow  # bug-patterns | design-decisions | workflow | constraints
---

Alternatives considered

Single c3spec memory command with interactive mode — prompts for category and slug. Deferred to later: the non-interactive form is sufficient for scripted retro workflows.

JSON/YAML memory store instead of markdown files — machine-readable. Rejected: markdown is the format agents read natively; agents already consume MEMORY.md as plain text.

Reason

promote solves the specific friction we identified: retro §6 candidates get checked in the markdown file but the memory file never gets created. The CLI command makes promotion a one-liner and enforces the invariant that a checked candidate has a corresponding file.

Risks / Trade-offs

RiskSeverityMitigation
Missed 'openspec' string after rename Medium Post-rename grep: grep -rn "'openspec'" src/ must return zero path-related hits. Verify separately from schema/format strings (some 'openspec' strings are format identifiers unrelated to directory paths — e.g. format: 'openspec' in the spec schema — leave those).
Init skills scaffold overwrites custom edits Low scaffoldC3specStructure checks for file existence before writing. Never overwrites existing files — idempotent re-runs are safe.
posthog-node removal breaks build Low No-op stubs export identical signatures. TypeScript compiler confirms compatibility before shipping.
c3spec memory promote mismatches slug patterns Low First version: exact string match on slug in retro files. If no match found, print a clear error with the expected pattern rather than silently succeeding.

Open Questions

All open questions resolved.

  • Format identifier strings: Rename all. format: 'openspec' → 'c3spec', author: 'openspec' → 'c3spec', metadata: { author: 'openspec' } → 'c3spec' everywhere. Complete brand excision — no openspec string survives in the source.
  • Legacy cleanup paths: src/core/legacy-cleanup.ts patterns (openspec-*.md, .claude/commands/openspec, etc.) — rename all to c3spec-*.md / .claude/commands/c3spec. These are the paths the CLI cleans up on upgrade; they need to match what c3spec actually installs.

Verification: after all changes, grep -rn "openspec" src/ must return zero results.
