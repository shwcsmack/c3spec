Remove Upstream Telemetry

Tier 2 proposal · May 26, 2026 · c3spec

Why

c3spec is a clean fork of upstream OpenSpec and should not carry a telemetry capability, PostHog history, or opt-out UX that implies events may be sent. The current implementation is already a no-op, but live specs, CLI hooks, tests, docs, and one archived PostHog change still describe telemetry as part of the product. This change removes telemetry as a first-class capability instead of preserving a disabled shell.

What Changes

Area From To Impact
CLI startup src/cli/index.ts imports telemetry hooks and calls notice, track, and shutdown functions. CLI startup keeps global color handling only; no telemetry import or lifecycle hook remains. Runtime simplification
Telemetry module src/telemetry/ exists as a disabled no-op plus config helpers. The telemetry module and its tests are deleted. Capability removed
Specs c3spec/specs/telemetry/spec.md requires PostHog behavior; global config requires telemetry fields. The telemetry spec is deleted; global config no longer mentions telemetry state. Spec removal
Historical archive c3spec/changes/archive/2026-01-09-add-posthog-analytics/ preserves the upstream PostHog change. The PostHog archive folder is deleted as part of complete excision. History prune
Docs and tests CLI docs and feedback/global config specs reference telemetry opt-outs and behavior. Live docs/specs/tests no longer reference telemetry, PostHog, or telemetry opt-out variables. Consistency cleanup

Capabilities

Removed

telemetry

The product no longer specifies, exposes, stores, or tests telemetry behavior.

Modified

global-config

Global config remains responsible for user-level settings and schema evolution, but not telemetry state.

Modified

cli-feedback

Feedback submission remains independent of environment and CI state, without mentioning telemetry settings.
