Clean Tree Gate Before Workflow Entry

Tier 2 Lightweight Feature - proposal - May 26, 2026

Why

c3spec workflows can currently start while the source checkout has tracked, uncommitted changes. That makes worktree setup and later cleanup noisy: the user discovers too late that unrelated edits need to be stashed, committed, or disentangled. The workflow should stop before any tier routing or worktree setup and ask the user how to handle the dirty source tree.

What Changes

Behavior Before After Impact
Workflow entry The agent can route into a tier and create a worktree while tracked changes are present. The agent checks git status --porcelain --untracked-files=no before tier work begins. Non-breaking workflow guardrail; prevents accidental carryover.
Dirty tree policy No consistent policy; cleanup happens ad hoc. Soft-block with three choices: stash and continue, commit first, or abort. Keeps the user in control while making the unsafe path explicit.
Direct tier entry Tier skills rely on worktree setup and do not guard the source repo. Each tier skill mirrors the same clean-tree pre-flight gate. Protects direct/resume paths, not only c3spec-start.
Untracked files Unspecified. Untracked files are ignored by the gate. Scratch files and local notes do not block normal work.

New Capabilities

workflow-clean-tree-gate

Defines the workflow requirement that c3spec checks for tracked uncommitted changes before starting routed tier work.

Skill-only first version

Implements the behavior in canonical skill instructions first, without introducing a new CLI helper. A future change can promote the check into code if instruction drift becomes a problem.

Source repo boundary

The gate applies to the checkout where the workflow starts, before creating a worktree. Newly-created worktrees still get their normal clean baseline checks.
