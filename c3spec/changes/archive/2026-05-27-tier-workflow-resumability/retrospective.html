1. Evidence

  • Commits on branch: lifecycle skill, tier skill alignment, continue/apply/host-adapter updates, source spec sync, generated host artifacts, focused contract tests, verification record.
  • Implementation tasks 1.1 through 5.5 are complete in tasks.md.
  • Verification: pnpm check:canonical-skills, pnpm run build, pnpm test (1415 tests), change/spec validation all passed.
  • Host generation: c3spec sync --force . regenerated tracked artifacts and repeat sync was stable.

2. What worked

  • Single lifecycle contract. Adding c3spec-tier-lifecycle as a reference skill gave tier, resume, apply, and archive helpers one shared source of truth instead of duplicating artifact rules in each skill.
  • Skills-first delivery. Most behavior landed in canonical skills with minimal code churn, which matched the approved design and kept CLI compatibility intact.
  • Focused contract tests. test/specs/tier-lifecycle-skill-contract.test.ts locked the important prose contracts without relying on more subprocess-heavy CLI tests.
  • Staged reviews caught real issues. Spec/quality review cycles surfaced concrete gaps (T1 status transitions, T2 pause gates, archive task-progress blocking, missing tier-skill lifecycle references) before they were committed.

3. What did not work

  • Initial skill drafts drifted from lifecycle semantics. Early tier1/tier2/archive wording needed multiple review passes for status transitions, optional HTML companions, and task-progress blocking.
  • Generated artifact tracking is uneven. c3spec sync writes the full Claude skill set, but only a subset is force-tracked in git; this made drift verification depend on sentinel/hash review rather than a simple git status view.
  • Spec terminology is still mixed in one area. Synced cli-artifact-workflow sections now say c3spec, while neighboring untouched scenarios still say openspec.

4. Workflow / process improvements

Observation Improvement
Tier skills and lifecycle contract evolved together Keep c3spec-tier-lifecycle as the first implementation task for any future tier-workflow change; treat tier skill edits as consumers, not co-owners of the contract.
Archive readiness needed explicit task-progress rules When changing lifecycle semantics, update archive helpers and their tests in the same stage as tier skill edits.
Host sync output is broader than tracked git paths Document the tracked-vs-ignored generated artifact convention in verification steps so repeat sync stability checks are interpreted correctly.

5. Learning worth capturing

Recommendation: capture a workflow learning that tier lifecycle behavior should stay centralized in c3spec-tier-lifecycle, with tier/resume/apply/archive skills acting as thin orchestrators over on-disk metadata (tier.md, required artifacts, status, and task progress) rather than re-deriving rules locally.

This generalizes beyond this change because future tier workflow edits will otherwise reintroduce contradictions between T1/T2/T3 skills, resume helpers, apply handoffs, and archive readiness checks.

Memory capture: recommended if the team wants this pattern preserved for future tier workflow work.
